<objective>
Add chapter header detection and schedule-aware chunking to the Act file splitter.

Purpose: The current section regex `/^(\d+[A-Z]*(?:-[A-Z]+)?)\.\s/gm` has two bugs: (1) CHAPTER headers between sections are silently dropped, losing chapter context; (2) numbered items inside schedules (e.g., "1. (1) The eligible fund...") produce 299 false section matches in act-2025.txt, splitting schedule content into phantom "sections" that collide with real Act section numbers.

Output: A new `splitActWithChaptersAndSchedules()` function that replaces `splitIntoSections()` for Act files — with chapter pre-pass, schedule boundary detection, and schedule-specific splitting.
</objective>

Create a new function `splitActWithChaptersAndSchedules(text: string): { section: string; text: string }[]` that replaces `splitIntoSections()` when processing Act files. Keep the old `splitIntoSections()` function in the file as a private helper (the new function calls it for the Act body portion).

**Step 1 — Schedule boundary detection:**
- Find the first SCHEDULE header in the text. Two formats (confirmed from research):
  - act-2025.txt: `SCHEDULE I`, `SCHEDULE II` ... `SCHEDULE XVI` -> regex: `/^SCHEDULE\s+[IVXLC]+\s*$/m`
  - act-1961.txt: `THE FIRST SCHEDULE`, `THE SECOND SCHEDULE` -> regex: `/^THE\s+\w+\s+SCHEDULE\s*$/m`
- Combined: `/^(?:SCHEDULE\s+[IVXLC]+|THE\s+\w+\s+SCHEDULE)\s*$/m`
- Split text into `actBody` (before first schedule header) and `scheduleBody` (from first schedule header onward)
- If no schedule header found, scheduleBody is empty and entire text is actBody

**Step 2 — Chapter-aware section splitting for Act body:**
- Detect CHAPTER headers: `/^(CHAPTER\s+[IVX]+(?:-[A-Z]+)?)\s*$/gm`
- The line AFTER a CHAPTER header is the chapter title (e.g., "COMPUTATION OF TOTAL INCOME")
- Track current chapter name as state while processing
- Run the existing section regex on actBody to get sections
- For each section, prepend chapter context to the section label: e.g., section "14" in CHAPTER IV becomes `"14 [Chapter IV — Computation of Total Income]"`
- If chapter title is very long (>60 chars), truncate with ellipsis
- Sections appearing before any CHAPTER header get no chapter annotation (preamble/preliminary sections)

**Step 3 — Schedule-specific splitting:**
- Create `splitScheduleArea(text: string): { section: string; text: string }[]`
- Split on SCHEDULE and PART boundaries: `/^(?:(?:THE\s+\w+\s+)?SCHEDULE\s*[IVXLC]*|SCHEDULE\s+[IVXLC]+|PART\s+[A-Z]+)\s*$/gm`
- Each segment between boundaries becomes one chunk with label from the boundary line (e.g., "Schedule I", "Schedule I -- Part A")
- Do NOT apply the section regex inside schedule text — this is the core bug fix
- If a schedule segment is longer than MAX_CHUNK_SIZE, use the existing subChunk() function to split it (with the schedule label as section name)
- If a schedule segment is very short (<50 chars), skip it (same as existing behavior)

**Step 4 — Wire it up:**
- In `buildChunks()`, when `config.splitter === 'act'`, call `splitActWithChaptersAndSchedules(text)` instead of `splitIntoSections(text)`
- The comparison splitter path remains unchanged
- All other code (subChunk, buildIndex, scoring, retrieval) is untouched

**Format for GST internal section IDs (locked decision — prep for Phase 8):**
- The section labels already include the source via the SourceConfig label in retrieveContext()
- For Phase 8 GST sources, the config.id will be 'cgst-2017' and section numbers will naturally avoid collision because they're scoped by source
- No special prefixing needed in this phase — the SourceConfig pattern handles it

**Testing consideration:**
- After the change, act-2025.txt should produce FEWER phantom sections (299 false matches removed) but MORE meaningful chunks (schedules now properly chunked + chapter annotations added)
- The net chunk count should be roughly similar or slightly different — the key metric is that schedule chunks have schedule labels (not section numbers)
